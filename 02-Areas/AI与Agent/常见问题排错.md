## 2026-07-03 Agent 死循环 — 反复调用同一个工具不停止

**场景**：Agent 在做多步推理时，反复调用 search 工具搜索同一个关键词，输出大量重复内容，最终耗尽 token 上限报错。

**根因**：
1. 没有设置最大推理步数限制（`max_steps`）
2. 工具返回的内容不包含"是否已足够"的判断信号
3. System Prompt 没有明确告诉 LLM "信息足够时就结束"

**结论**：
- 永远设置 `max_steps`（经验值 4-8 步，简单问题 3-4，复杂问题 6-8）
- 工具返回结果中可以加"无更多结果"等明确信号
- System Prompt 加上"如果你认为信息已经足够回答，请直接输出答案"

**关键代码**：
```python
# Agent 循环必须设置步数限制
max_steps = 6
for step in range(max_steps):
    # ...
    if msg.content and not msg.tool_calls:
        return msg.content  # LLM 认为可以回答了
raise TimeoutError(f"Agent 执行超过 {max_steps} 轮")

# System Prompt 约束
SYSTEM = """你可以调用工具获取信息。
当你认为信息已经足够回答问题时，直接给出答案。
调用工具的次数不要超过 5 次。"""
```

## 2026-07-03 Agent 答非所问 — 没有用 RAG 导致幻觉

**场景**：让 Agent 基于一份产品手册回答客户问题，Agent 给出了很多手册里没有的信息，部分还是错的。

**根因**：
1. 直接把长文档全文塞进 Prompt → 超出上下文窗口
2. 没有用向量检索，LLM 只能靠自己训练数据中的知识"脑补"
3. 没有明确要求"只基于提供的信息回答"

**结论**：
- 任何"基于文档问答"的场景，必须用 RAG 而不是全文塞入
- Prompt 中加"如果资料中没有相关信息，请明确说'根据现有资料无法回答'"
- 设置 `temperature=0` 减少创造性输出

**关键代码**：
```python
# RAG 正确做法
RAG_PROMPT = """基于以下参考资料回答问题。
如果参考资料中没有相关信息，请回答"根据现有资料无法回答"。

参考资料:
{context}

问题: {question}
回答:"""
```

## 2026-07-03 工具调错 — 给了搜索和计算工具，算数它却去搜索

**场景**：Agent 有 `search_web` 和 `calculator` 两个工具。问"125*37等于多少"，Agent 去搜索而不是计算。

**根因**：
1. 工具 `description` 太简略（`"搜索信息"`、`"计算"`）
2. 没有说清楚"何时用/何时不用"
3. 参数 description 不够具体

**结论**：
- 工具 description 必须写清楚使用场景和边界
- 给出正反例："何时使用" + "何时不使用"
- 用 `enum` 约束参数取值范围

**关键代码**：
```python
# ✅ 好的工具描述
{
    "name": "calculator",
    "description": """执行数学表达式计算。
    何时使用: 用户需要做数学计算、数值运算时。
    何时不使用: 查询数据、搜索信息（这些用 search_web）。""",
    "parameters": {
        "type": "object",
        "properties": {
            "expression": {
                "type": "string",
                "description": "数学表达式，如 '2*(3+4)'，数字和基本运算符"
            }
        }
    }
}
```

## 2026-07-03 向量检索不到关键信息

**场景**：把一份 300 页的技术规范文档导入 RAG，问了一个关键技术参数，Agent 说"根据资料无法回答"，但手动翻 PDF 确实能找到。

**根因**：
1. chunk 切得太大（5000 token），导致一个 chunk 包含太多信息，检索精确度下降
2. 没有用 semantic splitter，切分破坏了语义边界
3. 没有做重排序（Reranker），Top-3 结果可能不是最相关的

**结论**：
- chunk_size 控制在 500-1000 token
- 一定要加 overlap（100-200 token）
- 数据量大时（>1000 条）要用 Reranker 提升 Top-N 准确率
- 混合检索（向量 + BM25）比纯向量搜索准确率高 20-30%

## 2026-07-03 多轮对话中 Agent 忘了之前的上下文

**场景**：用户在第1轮说"我是一个 Java 后端工程师"，第5轮问"帮我优化这段 Java 代码"，Agent 反问"你用的是什么语言？"

**根因**：
1. 上下文窗口满了，最早的几轮对话被丢弃
2. 没有对话摘要机制
3. 关键信息（如用户背景）没有提取出来存入长期记忆

**结论**：
- 实现对话摘要：每 N 轮对话后，把前面的内容压缩成摘要
- 关键信息（个人背景、偏好、已确认的事实）写入向量数据库
- 每次新问题都先检索历史关键信息

## 2026-07-03 流式输出卡顿问题

**场景**：前端用 EventSource 接收 SSE 流式输出，但用户反馈"一顿一顿的，不是流畅输出"。

**根因**：
1. 前端收到每个 token 就立即渲染，太细粒度反而卡
2. 网络延迟导致 token 到达不均匀

**结论**：
- 前端加缓冲：攒够 4-8 个 token 再一次性渲染
- 后端控制推送频率：不要每个 token 都推，适当合并

**关键代码**：
```javascript
// 前端缓冲优化
let buffer = "";
let timer = null;

source.addEventListener("token", (e) => {
    buffer += e.data;
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
        display.textContent += buffer;
        buffer = "";
    }, 50); // 50ms 内收到的 token 合并渲染
});
```

## 2026-07-03 安装 Python 后命令不可用 — Microsoft Store 占位符冲突

**场景**：用 winget 安装了 Python 3.12，验证时 `python --version` 没反应（exit code 49），但实际 Python 已装好。

**根因**：
- Windows 的 Microsoft Store "App Execution Alias" 在 `PATH` 中优先级更高
- `C:\Users\Administrator\AppData\Local\Microsoft\WindowsApps\python.exe` 是指向 Store 的占位符
- 占位符在非交互式终端中直接退出，不执行真正的 Python

**结论**：
- Python 真实安装路径：`%LOCALAPPDATA%\Programs\Python\Python312\`（默认）
- 解决方案（二选一）：
  1. **推荐**: 在系统环境变量 PATH 中把 `%LOCALAPPDATA%\Programs\Python\Python312\` 移到 `WindowsApps` 之前
  2. **临时**: 用完整路径调用，如 `C:\Users\Administrator\AppData\Local\Programs\Python\Python312\python.exe --version`

**关键命令**：
```bash
# 验证真实 Python
"C:\Users\Administrator\AppData\Local\Programs\Python\Python312\python.exe" --version

# 设置国内 pip 源
pip config set global.index-url https://mirrors.aliyun.com/pypi/simple/
```

## 2026-07-04 RAG 检索不到关键信息 — 切分策略不对

**场景**：把一份 50 页的产品文档导入 RAG，问了一个具体参数值时，Agent 说"根据资料无法回答"，但手动在 PDF 里 Ctrl+F 确实能搜到。

**根因**：
1. chunk_size 设太大（2000 token），一个 chunk 里内容太多，向量被稀释
2. 没有 overlap，关键信息刚好在 chunk 边界处被截断
3. 只用向量检索（BM25 + 向量 混合检索更好）

**结论**：
- chunk_size 控制在 500-1000 token（中文信息密度高，可以稍大）
- chunk_overlap 至少 100 token（让边界信息两边都出现）
- 用混合检索（向量+BM25）替代纯向量检索，召回率提升 20-30%
- 对中文文档，separators 里一定要加"。" "！" "？"

**关键代码**：
```python
# 最好的默认切分配置
RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100,
    separators=["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""]
)
```

## 2026-07-04 Query 重写大幅提升检索准确率

**场景**：用户问"HashMap并发问题"，向量搜索直接搜"HashMap并发"，但文档里写的是"ConcurrentHashMap线程安全"，关键词对不上导致检索不到。

**根因**：
- 用户的自然语言和文档的书面语之间存在"语义鸿沟"
- 用户习惯口语化提问，文档使用专业术语
- 纯向量检索对简短的搜索词效果差

**结论**：
- 在检索前先用 LLM 重写 query："HashMap 并发问题" → "HashMap 线程安全 ConcurrentHashMap"
- 重写后的 query 和文档的匹配度显著提升
- 成本极低（一次 LLM 调用，每次 +0.001 元）

**关键代码**：
```python
def rewrite_query(question: str) -> str:
    """Query 重写：自然语言 → 搜索关键词"""
    response = client.chat.completions.create(
        model="deepseek-chat",
        messages=[{"role": "user", "content":
            f"将以下问题改写成适合搜索的关键词，只输出关键词:\n{question}"}],
        temperature=0
    )
    return response.choices[0].message.content
```
