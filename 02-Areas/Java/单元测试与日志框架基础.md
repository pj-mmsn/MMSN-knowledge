# 单元测试与日志框架

> **一句话**:测试和日志是"你觉得不重要但面试官觉得你不懂就是基本功不扎实"的东西。

## JUnit 5 + Mockito

### 基础结构

```java
@ExtendWith(MockitoExtension.class)  // JUnit 5
class OrderServiceTest {

    @Mock  // 模拟依赖
    private OrderDao orderDao;

    @InjectMocks  // 把 @Mock 的对象注入到被测对象
    private OrderService orderService;

    @Test
    @DisplayName("库存不足时应该抛异常")
    void shouldThrowWhenStockNotEnough() {
        // Given：准备数据
        when(orderDao.getStock(1L)).thenReturn(0);

        // When + Then
        assertThrows(InsufficientStockException.class,
            () -> orderService.createOrder(1L, 1L, 10));
    }

    @Test
    @DisplayName("正常下单成功")
    void shouldCreateOrderSuccessfully() {
        when(orderDao.getStock(1L)).thenReturn(100);

        Order order = orderService.createOrder(1L, 1L, 2);

        assertNotNull(order);
        assertEquals(1L, order.getUserId());
        verify(orderDao, times(1)).insert(any());  // 验证 insert 被调了1次
    }
}
```

### Mockito 核心方法

```java
// 打桩：模拟返回值
when(dao.findById(1L)).thenReturn(user);
when(dao.findById(anyLong())).thenThrow(new RuntimeException());

// 验证：检查方法是否被调用
verify(dao, times(1)).insert(any());
verify(dao, never()).delete(any());  // delete 从未被调用

// 参数匹配
any()           // 任意对象
anyLong()       // 任意 long
eq("expected")  // 精确匹配
argThat(u -> u.getName().equals("张三"))  // 自定义匹配

// @Spy：部分模拟（真实对象 + 部分方法模拟）
@Spy
private OrderService orderService;  // 大部分方法走真实逻辑
```

## SLF4J + Logback — Java 日志标准

```java
// SLF4J 是门面（接口），Logback/Log4j2 是实现
import lombok.extern.slf4j.Slf4j;

@Slf4j  // Lombok 自动生成 log 对象
public class OrderService {

    public void createOrder(Order order) {
        log.info("开始创建订单: orderId={}, userId={}", order.getId(), order.getUserId());
        //    ↑ 用 {} 占位符，不要用字符串拼接！
        //    ❌ log.info("开始创建订单: " + order.getId());  // 每次都拼字符串，浪费性能

        try {
            orderDao.insert(order);
            log.info("订单创建成功: {}", order.getId());
        } catch (Exception e) {
            log.error("订单创建失败: orderId={}", order.getId(), e);
            //                                                    ↑ 异常放最后，自动打堆栈
            throw new OrderException("创建订单失败", e);
        }
    }
}
```

### 日志级别

| 级别 | 含义 | 使用场景 |
|:--:|------|---------|
| ERROR | 错误 | 需要人工介入（支付失败、DB 挂了） |
| WARN | 警告 | 潜在问题（接近限流阈值、重试成功） |
| INFO | 信息 | **关键业务节点**（订单创建、用户注册） |
| DEBUG | 调试 | 开发调试（SQL 参数、方法入参出参） |
| TRACE | 追踪 | 最细粒度，一般不开 |

### Logback 配置要点

```xml
<!-- logback-spring.xml -->
<configuration>
    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 按天滚动文件 -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>  <!-- 保留 30 天 -->
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 按包名设级别 -->
    <logger name="com.xxx.order" level="DEBUG"/>      <!-- 自己项目的包 → DEBUG -->
    <logger name="org.springframework" level="WARN"/> <!-- Spring → 只打 WARN 以上 -->

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

## Spring Boot Test — 集成测试

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    @DisplayName("创建订单接口 — 正常")
    void shouldCreateOrder() throws Exception {
        mockMvc.perform(post("/api/v1/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content("""
                    {"userId": 1, "productId": 1, "amount": 99.00}
                    """))
                .andExpect(status().isCreated())
                .andExpect(jsonPath("$.code").value(0))
                .andExpect(jsonPath("$.data.id").isNotEmpty());
    }
}
```

## 面试话术

「单元测试我主要写 Service 层的核心逻辑，用的 JUnit 5 + Mockito。Mockito 的 when-verify 模式很好用。不是追求覆盖率 90%，而是让核心逻辑可回归——订单金额计算、状态流转这些，错了是生产事故。日志方面，生产上用 Logback，info 记录关键节点，error 记录异常并打堆栈，按天滚动保留 30 天。」
