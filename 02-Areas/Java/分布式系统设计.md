# 分布式系统设计

> **一句话**:CAP 是分布式理论基础（三选二），BASE 是互联网的实际方案（最终一致）。分布式事务靠 Seata/TCC/消息表。

## CAP 定理

```
网络分区发生时三选二：
  C (一致性) — 所有节点数据时刻一致
  A (可用性) — 每个请求都能响应
  P (分区容错) — 网络断开仍可用（P 不可避免）

  CP: ZooKeeper、etcd → 强一致，牺牲可用
  AP: Eureka、Cassandra → 高可用，最终一致
```

## BASE 理论

| 字母 | 含义 | 例子 |
|------|------|------|
| BA | 基本可用 | 双11只让看不让改 |
| S | 软状态 | 数据同步有中间态 |
| E | 最终一致 | 朋友圈晚 1 秒看到无所谓 |

## 分布式事务方案

| 方案 | 原理 | 适用 |
|------|------|------|
| 2PC | 两阶段提交 | 强一致，性能差 |
| TCC | Try-Confirm-Cancel | 资金类业务 |
| 本地消息表 | 业务+消息一起写，定时重试 | 最终一致 |
| **Seata AT** | 自动生成回滚 SQL | **最常用**，无侵入 |
| RocketMQ 事务消息 | Broker 回查生产者 | 阿里系 |

## 分布式 ID 方案

| 方案 | 说明 |
|------|------|
| UUID | 简单但无序，B+Tree 性能差 |
| 数据库自增 | 单点瓶颈 |
| **雪花算法** | 64bit（时间戳+机器ID+序列号），**推荐** |
| 美团 Leaf | 号段模式 + 雪花模式 |
